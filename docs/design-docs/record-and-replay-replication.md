# Record & Replay Replication

## 状态

本文记录 2026-06-26 形成的 Record & Replay 复刻方案。它是后续实现和评审的设计入口；证据细节看 `../references/codex-computer-use-reverse-engineering/record-and-replay-event-stream.md`，执行拆解看 `../exec-plans/active/20260626-record-and-replay-event-stream.md`。

当前方向是直接复刻官方 Record & Replay 的可观察行为，再补 OCU 为了独立运行必须拥有的控制条、等待和通知能力。官方兼容层和 OCU 扩展层必须分开维护。

“直接复刻官方”不是把所有 OCU 已实现的便利能力伪装成官方协议，也不是复制官方私有代码。这里的复刻边界是：用官方版本的可观察 MCP transcript、session 文件、`events.jsonl`、AX payload、截图上下文、control endReason 和 skill handoff 行为约束 OCU；未被官方样本确认的字段必须继续标记为 OCU baseline 或待校准。

## 目标

- 先在 macOS 上复刻官方 `record-and-replay` event stream 的 MCP surface、session 文件、事件语义、AX payload、compact diff 和截图上下文。
- 让 `open-computer-use event-stream mcp` 可以作为官方 `SkyComputerUseClient event-stream mcp` 的开源替代。
- 让独立 plugin / skill repo 只依赖 OCU runtime，不依赖 Codex.app 绘制 UI 或转发录制结束交互。
- 所有偏离官方的能力必须明确标记为 OCU 扩展，不能混进官方兼容 MCP surface。

## 已确定的方案口径

本轮决策用于约束后续实现：

- 第一阶段只聚焦 macOS，不把 Windows / Linux recording 纳入当前验收范围。
- 接受增加 `event-stream` 子命令；官方兼容入口是 `open-computer-use event-stream mcp`，独立集成入口是 `open-computer-use event-stream start/status/stop/cancel/wait/validate/summarize/scaffold-skill`。
- 目标是直接复刻官方可观察行为，再补 OCU 独立运行所需能力；不是先设计一套“更合理”的新协议，也不是让上层 skill/plugin 依赖聊天约定来解释录制产物。
- Codex.app 不作为 OCU 的运行依赖。即使官方宿主会提供 feature gate 或 MCP elicitation UI，OCU 也必须自己处理开始确认、录制控制条、Done / Discard、结束状态和 listener 唤醒。
- `cancel`、`wait`、notify callback、validate / summarize / scaffold 和 standalone repo 自检都只属于 OCU 扩展层；官方兼容 MCP surface 不因此增加工具或参数。
- AX compact diff、截图上下文、事件字段和 timeout endReason 都必须最终由 official golden recordings 校准；在此之前只能称为 OCU baseline。
- 独立 thin skill / plugin repo 的职责是包装和消费 OCU runtime，不复制录制实现，也不假设 Codex.app 私有 UI 或回调存在。

## 复刻与扩展的接口合同

后续实现和评审默认使用这份接口合同判断行为归属：

- **官方兼容合同**：`open-computer-use event-stream mcp` 的 server name、protocol、initialize response、`tools/list`、tool annotations、3 个 no-arg tools、session 文件布局、`events.jsonl` 主证据、`session.started` / `session.ended` 生命周期事件、官方已确认动作事件、AX payload、截图上下文和 skill handoff 都要向官方样本收敛。
- **OCU 扩展合同**：`event-stream cancel/wait/validate/summarize/scaffold-skill`、开始确认 UI、录制中控制条、wait listener、notify callback、`latest-session.json` / `active-session.json`、standalone repo 自检和 recording-to-skill scaffold 是 OCU 为独立运行提供的能力，可以比官方多，但必须在文档、脚本和输出字段中保持可识别。
- **校准合同**：没有 official golden recording 的字段或算法不能升级为“官方等价”。当前 AX compact diff、截图触发阈值、drag / terminal / selection / debug / secure input / raw event 字段、timeout endReason 和 MCP elicitation 业务 message / `_meta` 都仍在校准合同下。
- **独立 repo 合同**：后续真实 repo 应只包含 thin skill、runtime verifier、skill workflow verifier、package/self-check、synthetic recording-to-skill smoke 和可选 lifecycle smoke。默认 CI 不启动真实录制；真实桌面 lifecycle 只作为本地或授权环境里的 opt-in 检查。

## Baseline 版本定义

第一波 baseline 版本按“官方可观察行为优先 + OCU 独立运行补齐”定义，避免实现时把可用性、独立集成能力和官方等价混为一谈：

- **必须具备**：官方三件套 MCP surface、no-active response 对齐、session 文件模型、基础输入事件、AX full / compact diff baseline、按需截图上下文、开始确认、OCU 自有控制条、wait/notify、recording validation、summary、skill scaffold 和 standalone repo scaffold。
- **必须可验证**：默认 / matrix / screenshot / real-input action smoke、official surface compare、no-active compare、hostless raw timeout fixture、official capture preflight、OCU candidate pairing preflight、source standalone repo smoke、npm staged standalone repo smoke，以及 baseline summary artifact audit。
- **必须如实报告缺口**：required `simple-action-stop` official successful recording fixture 已入库并通过 readiness 后，baseline 可以把最小 required golden 标为完成；但 recommended `keyboard-input-stop`、`drag-stop`、`cancel`、`timeout` 未齐和算法级校准未完成时，仍必须报告 recommended gap，不能把全部 official successful recording schema 或 AX compact diff / screenshot 策略宣称为等价。
- **不得做的事**：不得把 `cancel`、`wait`、notify callback、validation、summary 或 scaffold 参数加入 `event_stream_start/status/stop` 的 tool schema；不得依赖 Codex.app 私有 UI 绘制录制条或转发 Done / Discard；不得把 hostless raw timeout 或 no-active response fixture 当成 successful recording 证据。

## 后续推进基准

后续基于这份方案继续做 baseline 或拆独立 repo 时，默认按下面顺序推进：

- **先复刻官方 successful recording**：required `simple-action-stop` official golden 已通过正常 Codex hosted Record & Replay 流程采集和脱敏入库；之后继续补 `keyboard-input-stop`、`drag-stop`、`cancel` 和 `timeout`。采集包只负责准备 wrapper 和占位输入，不启动官方录制；正式入库必须先跑 inspect-only，再脱敏导入，再跑 strict audit gate 留存 summary artifact。
- **再做 same-scenario OCU candidate**：official fixture 入库后，用 `scripts/ingest-ocu-record-and-replay-candidate.py` 生成同 scenario candidate。candidate 导入不作为 required official golden strict gate 的硬条件；它会输出 `fixtureSetGateShell` / `pairingPreflightShell`，需要校准 OCU 与官方事件形态时再显式运行集合级强对比。
- **fixture set gate 的 candidate compare 默认走强策略**：同 scenario compare 必须要求 AX diff evidence、AX diff marker 一致、suppressed event sequence 一致和 suppressed schema 一致；如果官方样本没有 diff 或 suppressed stream，gate 应显示对应强策略已打开但不臆造官方字段。这样后续官方样本一旦出现 compact diff 或 suppressed fallback，OCU candidate 缺失或漂移会直接失败。真实 hosted official 样本会包含录制控制条、结束点击等宿主事件，synthetic OCU candidate 可能天然不具备相同事件序列，因此 candidate compare 是校准/差距分析门，不是 required official golden 完成条件。
- **OCU 扩展只补独立运行体验**：开始确认 UI、录制条、Done / Discard、`wait --session-id`、notify callback、standalone repo verifier 和 recording-to-skill scaffold 都留在 OCU 扩展层；它们可以是 baseline 可用性的必要条件，但不能作为 official successful recording 等价证据。
- **严格区分两个完成状态**：`usableBaseline=true` 只说明当前 OCU / standalone baseline 可用；只有 required official successful recording 覆盖和 readiness 都通过后，才能把 `officialSuccessfulRecordingEquivalenceReady` 置为 true。same-scenario OCU candidate compare 的结果继续作为校准 evidence 单独解读。

## 已确认的官方事实

- 官方 `record-and-replay` 插件启动的是 `Codex Computer Use.app` 里的 `SkyComputerUseClient event-stream mcp`。
- 官方普通 Computer Use 和 Record & Replay 共用同一个 Computer Use runtime，只是 MCP mode 不同。
- 当前对 Codex.app asar 的观察只确认了 `recordAndReplay` feature gate、插件可见性和通用 MCP elicitation / approval panel 逻辑；没有证据表明 Codex.app 自己实现 `event_stream_*` tools、写 recording files 或绘制录制控制条。
- 官方 Record & Replay MCP server name 为 `Record & Replay`，协议版本为 `2025-11-25`。
- 官方 MCP surface 只有 3 个无参数 tools：`event_stream_start`、`event_stream_status`、`event_stream_stop`。
- 官方 skill 的 concurrent recording 语义是“一次只允许一个 active recording”：`event_stream_start` 如果报告已有 active recording，agent 不应静默当作新的 demonstration 继续推进，而应告知用户已有录制正在进行，并询问要使用这个 active recording 还是等它结束。
- 官方 start tool description 和 skill 流程都要求录制最长 30 分钟；agent 在 start 成功后应结束当前 turn、请用户完成后回来，并明确告知这个时限。
- 官方 skill 把 `events.jsonl` 当主输入，`metadata.json` 或 `session.json` 提供 timing、路径和状态。
- 官方 strings 确认了 `events.jsonl`、`metadata.json`、`suppressed.jsonl`、`session.json`、`eventsPath`、`metadataPath`、`suppressedEventsPath` 等产物或字段。
- 官方 strings 确认录制控制条文案和结束原因：`recording_controls_stopped`、`recording_controls_cancelled`。
- Required `simple-action-stop` official successful recording fixture 已入库：`docs/references/codex-computer-use-reverse-engineering/fixtures/recordings/official-simple-action-stop-1.0.857/`，可用于校准最小 stopped click scenario、hosted response-shape evidence 和 `recording_controls_stopped` handoff。
- 官方 strings 确认事件名至少包括 `session.started`、`mouse.click`、`mouse.context_menu`、`mouse.drag`、`keyboard.text_input`、`keyboard.submit`、`keyboard.shortcut`、`terminal.value_changed`、`selection.changed`、`AX.focusedWindowChanged`、`window.changed` 和 `debug.error`，并出现过 `secureInput` 线索。
- 官方 AX compact diff 使用 `~`、`+`、`-` 表示 changed、added、removed，并存在 diff 过大时回退或跳过 diff render 的行为。
- 官方 strings 确认 `screenshotNeededForContext` 和 `accessibilityInspectorPayload`，但没有证据表明每个事件都默认截图。
- 当前没有确认独立 scroll event 名称，也没有确认 timeout 对外 endReason。

## 官方兼容层

这一层只服务于“像官方一样被 MCP host 使用”。

- `event-stream mcp` 只暴露 `event_stream_start`、`event_stream_status`、`event_stream_stop`。
- 这 3 个 tools 保持无参数，不加入 `cancel`、`wait`、callback、webhook 或 OCU 私有参数。
- `start` 已有 active session 时返回当前 session，不新建并行录制；agent-facing skill 应把这个结果解释为需要用户选择“使用当前录制”或“等待当前录制结束”，不能静默把它当成本次新录制。
- `status` 返回当前或最近 session，必须包含可读取的 metadata 和 events 路径。
- `stop` 幂等；无 active session 时返回最近可解释状态。
- session 文件至少包含 `events.jsonl`、`metadata.json`、`session.json` alias 和 `suppressed.jsonl`。
- `events.jsonl` 是后续生成 skill 的事实来源，metadata 只提供会话边界、计数、路径和 endReason。
- 事件 schema、字段名、顺序、timing 和 AX payload 以 official golden recordings 为最终校准依据。

## OCU 扩展层

这一层只服务于“脱离 Codex.app 后仍能独立运行”。

- OCU 必须自己承载开始确认 UI、录制中控制条、Done / Discard、`wait` 唤醒和 notify callback；独立 plugin / skill 不应依赖 Codex.app 替它画 bar 或转发录制结束交互。
- CLI 扩展保留在 `open-computer-use event-stream start/status/stop/cancel/wait --json`。
- `cancel` 只作为 OCU CLI/runtime 能力，不暴露为官方兼容 MCP tool。
- `wait` 是第一版 listener 模型：调用方启动录制后阻塞等待，用户点击 Done / Discard 或外部 stop/cancel 后返回最终 status；`wait --json` 额外带 `waitTimedOut` 和 `waitSessionMatched` 作为 OCU 扩展字段，供独立集成方区分结束唤醒、超时返回和 session id 不匹配。
- `wait --session-id <id>` 会优先读取对应 session 目录里的完成态 metadata，因此即使 latest session 已被后续录制覆盖，也能立即返回目标历史 session；如果指定的 session id 不存在或当前 active session 明显不匹配，会返回 `waitTimedOut=true` / `waitSessionMatched=false`，避免独立集成方无限等待错误 id。
- `wait --notify-command '<json-argv>'` 是本地 callback 模型：只在 `waitTimedOut=false` 时同步执行，stdin 写入最终 status JSON，同时通过 `OPEN_COMPUTER_USE_EVENT_STREAM_*` 环境变量暴露 session、state、endReason、metadataPath、sessionPath、eventsPath 和 suppressedEventsPath；callback 默认 10 秒超时，失败会让 CLI 返回 error。

## Start Approval 与录制 UI

官方 strings 能确认 `Record & Replay approval cancelled via MCP elicitation.` 和 `Record & Replay approval denied via MCP elicitation.`；`Codex.app` asar 中也能看到通用 MCP elicitation/approval panel 逻辑。但当前可观察证据没有显示 `Codex.app` 实现 `event_stream_*` tool 或绘制录制控制条。更合理的边界是：

- `Codex.app` 是宿主，负责 feature gate、插件可见性和通用 MCP elicitation panel。
- `Codex Computer Use.app` / event-stream runtime 负责录制能力、控制条、事件文件和结束原因。
- 官方 Record & Replay skill 在自然语言层面要求“用户准备好后再调用 start”，runtime 侧也存在 approval cancelled / denied 的错误语义。

OCU 需要同时满足两类使用方：

- 官方兼容 MCP host：仍然只看到 `event_stream_start/status/stop` 三个无参数 tools。以后如果实现 MCP elicitation，应作为 start 内部流程，不改变 tool schema。
- 独立 plugin / skill / CLI 使用方：不能依赖 Codex.app 的 approval UI，所以 OCU 应提供自有的开始确认 UI 或可配置 approval policy，并让控制条、Done / Discard 和 `wait` 都直接落到 OCU runtime。

推荐实现顺序：

1. 保持现有默认自动化路径可跑，测试和 smoke 可关闭 start approval。
2. 在 `EventStreamService.start()` 前增加可替换的 start approval gate，先沉淀 approved / denied / cancelled 三种结果和官方错误文案。
3. 在 macOS interactive runtime 中接入 OCU 自有开始确认浮层；用户确认后才创建 session 和安装事件 monitor，用户拒绝或取消时不写 session。开始确认文案应告知录制时限，默认是官方一致的 30 分钟上限。
4. MCP elicitation baseline 已接入：在线性 stdio server 内补 server-initiated request/response 支持，只在支持 `capabilities.elicitation` 的 host 上启用；不支持时回退到 OCU 自有 UI 或配置策略。当前已按 Codex.app host schema 使用显式 `mode: "form"` + 空对象 schema，真实官方 Record & Replay 业务 message / `_meta` 仍待 golden 样本校准。
5. 把 approval outcome 写入 status / error，但不要暴露新的官方 MCP tool，也不要把 host-specific callback 放进官方兼容层。

## 实现分层

### Runtime

`EventStreamService` 是录制状态机，负责 active session、文件写入、事件 append、结束原因和 listener 唤醒。它必须运行在持久进程内；短生命周期 CLI 只能做调试，不能作为真实录制状态源。

关键约束：

- start/status/stop/cancel/wait 通过 app-agent 访问同一份进程内状态。
- `wait` 不能阻塞其它连接的 stop/cancel。
- `finish()` 必须先写最终 `session.ended` 和 metadata，再暴露 inactive 状态。
- 权限缺失、AX snapshot 失败、截图失败、事件压缩失败和 Input Monitoring monitor 安装失败都应可恢复，并进入 status、主事件的低敏 `debug.error` 或 suppressed stream。

### Event Stream

第一阶段必须覆盖官方已确认事件名：

- session：`session.started`、`session.ended`
- mouse：`mouse.click`、`mouse.context_menu`、`mouse.drag`
- keyboard：`keyboard.text_input`、`keyboard.submit`、`keyboard.shortcut`
- terminal：`terminal.value_changed`
- selection：`selection.changed`
- window：`window.changed`
- AX：`AX.focusedWindowChanged`
- debug：`debug.error`

OCU baseline 已有点击、右键、拖拽、文本输入、提交、快捷键、terminal value changed、selection changed、window changed、debug error、experimental raw events 和 AX focused window。`mouse.drag` 当前按 mouse down / dragged / up 的起止位置、距离和时长记录，字段结构仍需 official golden recordings 校准。`terminal.value_changed` 当前只在明显终端 app / 终端 focused element 且 AX value 变化时记录，payload 写 `valueHash`、`valueLength` 和脱敏后的 `focusedAccessibilityElement`，不写终端缓冲区全文；事件触发时机、字段名和官方是否记录明文仍需 official golden recordings 校准。`selection.changed` 当前在 focused AX element 暴露非空 `selectedText` 且 selection signature 变化时记录选择内容；如果此前已记录过选择而当前选择为空，也会写 `selectionCleared=true` 的清空事件，避免后续 skill 误用 stale selection；字段名、触发时机和清空选择时的官方行为仍需校准。`window.changed` 当前在录制开始和输入事件前 app/window context 变化时写入，并继续补 `AX.focusedWindowChanged` 作为上下文；字段名、是否初始帧也写入、以及与 AX event 的顺序仍需 official golden recordings 校准。`debug.error` 当前在 AX snapshot、AX diff cumulative fallback、screenshot 写入失败和 Input Monitoring monitor 全部安装失败这类上下文采集降级时写入主事件流，payload 只保留 `subsystem`、`reason`、`errorType` 和低敏 context；Input Monitoring 不可用时会写 `subsystem=inputMonitoring` / `reason=inputMonitorsUnavailable`，避免独立集成方把没有输入事件的录制误判为正常空录制；完整错误细节仍进入 `suppressed.jsonl`，字段仍需官方样本校准。`secureInput` 当前作为安全 baseline：focused AX element 暴露 protected/secure/password 信号时，键盘文本事件只写 `secureInput=true` 和长度，不写明文 `text`，raw key event 也只写 redaction 标记和字符数；字段仍需官方样本校准。`experimentalRawEvents` 当前作为低层 NSEvent 摘要附在点击、拖拽和键盘事件上；scroll wheel 在没有确认官方独立 scroll event 名称前只写 raw-only `experimentalRawEvents` 事件，且 opt-in action smoke 已用真实 CG scroll wheel 验证该 raw-only 路径会入流。为了让生成 skill 不漏掉滚动意图，summary/scaffold 会把 `reason=scrollWheel` 的 raw-only event 当作 replay action 渲染为 Scroll step；这仍是 OCU baseline，待官方 scroll schema 校准。

### AX Payload

AX payload 是复刻质量的关键，不应长期停留在每次 full tree。

- 首帧或无法 diff 时输出 full tree。
- 同一 app/window 的后续 payload 输出 compact diff。
- compact diff 使用官方风格的 `~`、`+`、`-`。
- diff payload 保留 previous diff 作为主 `renderedText/treeLines`，并额外输出从 initial tree 到 current tree 的 cumulative diff 字段。
- diff 过大、removed summary 超预算或 baseline 不稳定时回退 full tree，并记录 suppressed reason。
- 后续用 official golden fixtures 校准 element identity、line budget、removed range summary、cumulative diff 和 fallback 条件。

### Screenshot Context

截图是上下文补充，不是默认每事件主数据。

- 默认策略应最小化截图采集。
- 当前 OCU 提供 `OPEN_COMPUTER_USE_EVENT_STREAM_SCREENSHOTS=auto|always|never` 供验证和调试。
- payload 标记 `screenshotNeededForContext` 和 `screenshotAvailable`，写入成功时提供 `screenshotPath`。
- 官方触发阈值、路径字段和 suppressed 行为仍待 golden recordings 校准。

### Recording Controls

控制条属于 OCU 独立性要求，不属于官方 MCP surface 扩展。

- Done 结束为 `recording_controls_stopped`。
- Discard 结束为 `recording_controls_cancelled`。
- 控制条必须能在普通窗口、多显示器、Space 切换和权限缺失场景下给出明确结果或错误。
- 自动化和 headless 验证可关闭控制条。

## 推进顺序

1. 固化官方 golden recordings：stop、cancel、重复 start、无 active status/stop、最长时长或模拟 timeout；入库时同时带上脱敏 MCP transcript，先跑结构 validator，再跑 golden readiness gate，确认动作类样本有一致的 `metadata.json` / `session.json` alias、可信的 metadata event/suppressed counts、唯一且位于首事件的 `session.started` 和唯一且位于最终事件的 `session.ended`，并在样本用于校准 AX / screenshot / 压缩 fallback 时显式要求 suppressed evidence，最后才进入 OCU candidate compare。
2. 对齐 MCP response 和 session 文件 schema：字段名、状态流转、endReason、路径命名、idempotent 行为。
3. 补 start approval gate：先沉淀 denied / cancelled 语义，再接 OCU 自有开始确认 UI，最后评估是否实现 MCP elicitation。
4. 对齐事件 schema：鼠标、键盘、AX、drag、raw events，确认是否存在 scroll event。
5. 校准 AX compact diff：从当前轻量实现推进到官方样本可比较的算法级行为。
6. 强化 OCU 扩展层：控制条场景、wait listener、notify callback、后续订阅模型和权限错误。
7. 拆独立薄 plugin / skill repo：只包装 OCU runtime，不复制录制逻辑；以 scaffold 生成的 `record-and-replay-skill-repo.json` 为 contract，保持 `checks` / `optionalChecks` 分层，发布前跑默认 `scripts/check.sh`，有桌面权限时再显式跑可选 lifecycle smoke。
