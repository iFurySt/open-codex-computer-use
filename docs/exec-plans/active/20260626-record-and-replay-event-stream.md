# Record & Replay Event Stream

## 目标

在 `open-codex-computer-use` 内复刻官方 Record & Replay 的 macOS event-stream 能力，优先对齐官方 `record-and-replay` 1.0.857 的 MCP surface、文件产物和事件语义，再补充独立于 Codex.app 的录制控制条、等待命令和结束通知能力。后续可以把薄插件和 skill 单独放到一个 repo，只依赖本仓库提供的 runtime。

## 范围

本阶段只覆盖 macOS。目标是让使用方可以通过 `open-computer-use event-stream mcp` 获得官方兼容的 3 个 MCP tool，并可以通过本仓库扩展 CLI 控制录制生命周期。

包含：

- 官方兼容 MCP tools：`event_stream_start`、`event_stream_status`、`event_stream_stop`，保持无参数 surface。
- 本地 CLI 扩展：`event-stream start/status/stop/wait/validate/summarize --json`，用于非 MCP 调用方和独立 skill/plugin 集成。
- Codex 安装便利命令：`install-codex-record-and-replay-mcp`，写入独立 `record_and_replay` server，启动 `open-computer-use event-stream mcp`。
- 录制产物：`events.jsonl`、`metadata.json`、`session.json`、`suppressed.jsonl`，以及 status 中的路径字段。
- 事件采集：鼠标、键盘文本输入、AX 事件、app/window attribution、截图上下文标记。
- 官方风格 AX compact diff：支持 full tree、diff from previous、cumulative diff，并保留必要的降级策略。
- OCU 自有录制控制条：显示录制中、可拖拽、Stop/Cancel、计时，并由 OCU 自己接收交互。
- 等待和通知：录制结束后唤醒 `wait` 调用方，供上层 skill 或插件继续生成后续资产。

不包含：

- Windows / Linux 录制能力。
- 与 Codex.app 私有 UI、feature flag 或插件宿主耦合。
- 直接复制官方私有实现代码。
- HTTP webhook 或复杂多 listener 协议；当前第一版已包含本地 `wait --notify-command` callback baseline，但它属于 OCU 扩展层，不进入官方兼容 MCP surface。

## 背景

官方 `record-and-replay` 插件本质上仍依赖 `Codex Computer Use.app` 内的 `SkyComputerUseClient`，只是以 `event-stream mcp` 子命令启动另一组 MCP tools。逆向上下文已沉淀在：

- `docs/references/codex-computer-use-reverse-engineering/record-and-replay-event-stream.md`
- `docs/references/codex-computer-use-reverse-engineering/fixtures/record-and-replay-event-stream-surface-1.0.857.json`
- `docs/design-docs/record-and-replay-replication.md`

关键事实：

- 官方插件路径为 `openai-bundled/record-and-replay/1.0.857/`。
- 官方 `.mcp.json` 启动命令为 `SkyComputerUseClient event-stream mcp`。
- 官方普通 `computer-use` 插件使用同一个 client，只是参数为 `mcp`。
- `Codex.app` asar 里只观察到 `recordAndReplay` feature flag、插件可见性和通用 MCP elicitation/approval 逻辑，没有发现 `event_stream_*` tool 实现或录制控制条文案。
- 录制控制条、`event_stream_*` tools、`events.jsonl` / `metadata.json` / `suppressed.jsonl` 字符串均出现在 `Codex Computer Use.app` 相关二进制中。
- 官方二进制字符串包含 `Record & Replay approval cancelled via MCP elicitation.` 和 `Record & Replay approval denied via MCP elicitation.`，说明 start 前 approval 有 runtime 可观察错误语义；但独立 OCU 不能依赖 Codex.app 的通用 panel，因此需要自有 start approval gate 或 UI。
- 当前已保存官方 1.0.857 的 `initialize` / `tools/list` normalized fixture；它不包含录制输出，仍不能替代后续 official golden recordings。
- 当前已新增 raw probe 脚手架 `scripts/probe-event-stream-recording.py`；本机官方 1.0.857 在宿主外可返回 initialize / tools-list，但 raw `event_stream_start` / `event_stream_status` / `event_stream_stop` 仍超时，没有产出 official recording paths。该 timeout transcript shape 已脱敏保存为 `docs/references/codex-computer-use-reverse-engineering/fixtures/record-and-replay-official-raw-start-timeout-1.0.857.json`。
- 2026-06-27 通过 Codex-hosted Record & Replay MCP tool 观察到 no-active `event_stream_status` / `event_stream_stop` 的 text JSON 都是 `{"isRecording": false, "maxDurationSeconds": 1800}`，没有 `state` / `active` / path 字段；该 response shape 已保存为 `docs/references/codex-computer-use-reverse-engineering/fixtures/record-and-replay-official-no-active-status-stop-1.0.857.json`，并用于校准 OCU MCP idle/no-active 返回。

## 方案

### 1. 固化官方样本

先用当前安装的官方 1.0.857 作为 baseline，采集最小录制样本：

- 正常 stop：点击、键盘输入、窗口切换各至少一次。
- cancel：通过官方控制条取消一次。
- 无 active session 时调用 status/stop。
- active session 时重复 start。

样本要作为 golden fixtures 保存到 `docs/references/` 或 `test-fixtures/` 的脱敏目录中，记录官方版本号、采集日期和脱敏方式。如果后续需要拉取 LFS 最新 bundle，应保留版本化对比，而不是覆盖 baseline。

### 2. 增加 event-stream 子命令

在现有 CLI 分层上增加：

- `open-computer-use event-stream mcp`
- `open-computer-use event-stream start --json`
- `open-computer-use event-stream status --json`
- `open-computer-use event-stream stop --json`
- `open-computer-use event-stream cancel --json`
- `open-computer-use event-stream wait --json [--session-id ...] [--timeout ...] [--notify-command '<json-argv>']`
- `open-computer-use event-stream validate --json [--strict-ocu] [--require-event-type ...] [--require-skill-draft] <session-dir-or-metadata-or-events>`
- `open-computer-use event-stream summarize --json [--include-text] [--require-action] <session-dir-or-metadata-or-events>`
- `open-computer-use event-stream scaffold-skill --json --skill-name <name> --output-dir <dir> [--description <text>] [--overwrite] [--include-text] <session-dir-or-metadata-or-events>`

MCP 模式只暴露官方兼容 3 tools。扩展命令留在 CLI，不污染官方兼容 surface。

### 3. 建立 session 文件模型

建议每次录制创建独立 session 目录，包含：

- `events.jsonl`：append-only 事件流，作为主产物。
- `metadata.json`：session id、start/end 时间、endReason、event count、路径和状态。
- `session.json`：与 `metadata.json` 内容一致的兼容 alias；官方 skill 文案和二进制字符串都提到过 `metadata.json or session.json`。
- `suppressed.jsonl`：被限流、合并、隐私策略或体积策略丢弃的事件。
- `screenshots/`：仅在 `AX.focusedWindowChanged` payload 判断需要额外视觉上下文且当前 snapshot 有截图数据时写入的 PNG，不作为每个事件的默认产物。
- `latest-session.json` / `active-session.json`：OCU 扩展索引文件，供 CLI `status` / `wait` 在非 MCP 场景下查找当前或最近 session。

`event_stream_start` 如果已有 active session，应返回当前 active session，而不是新建。agent-facing skill 不能把这个结果静默当成本次新 demonstration；它应该告知已有录制正在进行，并询问用户要使用这个 active recording 还是等待它结束。`event_stream_status` 返回当前或最近一次 session。`event_stream_stop` 要 idempotent，无 active session 时返回最近状态。

`wait` 用作 OCU 扩展 listener：通过 app-agent 访问长驻 recorder 状态时，`status/stop/cancel/wait` 不应持有全局环境覆盖锁，避免一个长时间 `wait` 阻塞另一个连接上的 `stop` / `cancel`。`finish()` 写最终 metadata 前也不能先清空 active session，否则并发 `wait` 可能读到旧的 active metadata 并误报 `recording_process_unavailable`。`wait --json` 返回体额外包含 `waitTimedOut` 和 `waitSessionMatched`，这是 OCU 扩展字段，不属于官方兼容 MCP surface。`wait --session-id <id>` 会优先读取目标 session 目录里的完成态 metadata，即使 latest session 已被后续录制覆盖，也能返回指定历史 session；如果指定 id 不存在或当前 active session 明显不匹配，会返回 `waitTimedOut=true` / `waitSessionMatched=false`，避免独立集成方无限等待错误 id。`wait --notify-command '<json-argv>'` 也是 OCU 扩展：仅当 `waitTimedOut=false` 时同步执行本地回调，stdin 写最终 status JSON，并通过 `OPEN_COMPUTER_USE_EVENT_STREAM_*` 环境变量暴露 session、state、endReason、metadataPath、sessionPath 和 eventsPath；回调默认 10 秒超时，可用 `OPEN_COMPUTER_USE_EVENT_STREAM_NOTIFY_TIMEOUT_SECONDS` 覆盖，失败会让 CLI 返回 error 并在 JSON 里写入 `notification`。`validate --json`、`summarize --json` 和 `scaffold-skill --json` 是本地只读/文件生成命令，不通过 app-agent 代理；前者用于检查 session 文件一致性、required event type，以及可选 `--require-skill-draft` 的 skill 草稿最低证据门槛，中者用于生成 skill 前快速审计事件流，后者把可审计 summary 写成第一版 `SKILL.md` 草稿和脱敏 `references/recording-summary.json`。`--require-skill-draft` 要求至少一个高层动作、recording 已完成、未取消、`session.started` 只出现一次且为第一条事件、`session.ended` 只出现一次并收尾，且没有 `inputMonitorsUnavailable` 等阻断性诊断；active recording、`active=true`、缺少 `session.started`、`session.started` 非首事件或缺少 `session.ended` 的 event stream 都会被拒绝。它允许其它 `needsReview` 的 recording 进入草稿生成，避免把尚待官方校准的 AX / screenshot 细节误设成硬门槛。`sessionPath` 是 OCU 便利字段，指向 `metadata.json` 的 `session.json` alias；官方兼容层仍以 `metadataPath` / `eventsPath` 为主。

### 4. 复刻事件语义

第一版事件类型优先覆盖官方已观察字符串：

- `session.started`
- `mouse.click`
- `mouse.context_menu`
- `mouse.drag`
- `keyboard.text_input`
- `keyboard.submit`
- `keyboard.shortcut`
- `terminal.value_changed`
- `selection.changed`
- `AX.focusedWindowChanged`
- `debug.error`
- AX tree full snapshot
- AX tree compact diff

每个事件应带 timestamp、app/window attribution、必要的 AX context，以及是否需要截图上下文的标记。截图不应默认每个事件都采集，应按官方线索使用 `screenshotNeededForContext` 类字段触发。

OCU 当前输入事件上下文：

- `mouse.click` 写入 y-down screen-state `location`，同时保留原始 AppKit `appKitLocation` 供后续校准。
- `mouse.click` 会尽量通过 `AXUIElementCopyElementAtPosition` 写入 `targetAccessibilityElement`，包含 role/title/label/value/selectedText/frame/windowFrame/actions 等轻量字段。
- 右键点击按官方字符串记录为 `mouse.context_menu`，同时保留 `button=right` 和目标 AX 摘要。
- `keyboard.text_input`、`keyboard.submit`、`keyboard.shortcut` 会尽量从 system-wide focused element 写入 `focusedAccessibilityElement`，用于识别文本输入、提交和快捷键目标。
- 官方字符串中还出现 `window.changed`、`selection.changed`、`mouse.drag`、`terminal.value_changed`、`debug.error`、`secureInput` 和 `experimentalRawEvents`；当前 baseline 已补 `window.changed`、`mouse.drag`、轻量 terminal value changed、轻量 selection changed 与 selection clear、低敏 debug error、Input Monitoring monitor 全部安装失败时的 `inputMonitoring/inputMonitorsUnavailable` debug event、secure input 脱敏和低层 raw event 摘要，但 window / drag / terminal / selection / debug / input monitoring failure / secure input / raw 字段结构仍需 official golden recording 校准。没有观察到明确 scroll event 名称，OCU 暂时只把 scroll wheel 写成 raw-only `experimentalRawEvents` 事件，不臆造 scroll event type。
- target/focused summary 只作为可消费上下文；命中失败不阻断录制。

OCU 当前截图策略：

- `OPEN_COMPUTER_USE_EVENT_STREAM_SCREENSHOTS=auto|always|never` 控制 Record & Replay 截图上下文，默认 `auto`。
- `always`：只要 snapshot 带有 screenshot PNG，就为 AX payload 写入 `screenshots/<timestamp>-<payload-kind>-context.png`。
- `never`：只保留 AX payload，不写截图。
- `auto`：当 AX tree 很短或缺少 focused summary 时认为上下文不足，标记 `screenshotNeededForContext=true` 并尝试写截图。
- payload 会包含 `screenshotNeededForContext`、`screenshotAvailable`，写入成功时额外包含 `screenshotPath`。
- 如果策略要求截图但当前没有截图数据，或写文件失败，事件不中断；写文件失败会进入 `suppressed.jsonl`。

### 5. 复刻 AX compact diff

实现目标是可用官方样本做 golden comparison，而不是只实现概念相似的 diff。

已观察到的官方 diff 语义：

- `diffFromPrevious`
- `DifferenceBaseline`
- `UIElementRenderDifference`
- `UIElementRenderDifferenceBuffer`
- `The following is a diff from the previous accessibility tree`
- `The following is a cumulative diff from the initial accessibility tree`
- compact render 使用 `~`、`+`、`-` 表示 changed / added / removed。
- diff 过大或 removed element summary 超预算时，官方会跳过 diff render 并回退。

OCU 实现应先复用现有 AX snapshot renderer，再增加稳定 element identity、tree diff 和 line budget。无法稳定 diff 时回退 full tree，并在 metadata 或 suppressed stream 中记录原因。当前 baseline 已输出 previous diff，并额外输出从 initial tree 到 current tree 的 cumulative diff 字段；字段名、line budget 和算法仍待 official golden recording 校准。

### 6. 增加 OCU 自有控制条

因为本仓库不能依赖 Codex.app 替我们绘制录制 UI，macOS runtime 需要自己绘制浮层：

- 显示 “Record & Replay is recording your actions” 或等价文案。
- 提供 Stop、Cancel、timer、拖拽移动。
- Stop 结束为 `recording_controls_stopped`。
- Cancel 结束为 `recording_controls_cancelled`，并保留 cancellation 状态供上层 skill 识别。
- 控制条点击直接进入 OCU runtime，不通过 Codex.app。

控制条应只在交互式录制时显示；headless 或 fixture 模式允许关闭。

### 7. 补 start approval gate

官方 skill 在流程上要求用户准备好后才调用 `event_stream_start`，官方 runtime 字符串也确认存在 approval cancelled / denied 错误。OCU 需要把这个流程落在自己这边，但不能改官方兼容 MCP surface。

第一阶段建议做成 runtime 内部 gate：

- `event_stream_start` 仍保持无参数。
- 已有 active session 时直接返回 active status，不重复弹 approval。
- 无 active session 时先进入 approval gate，只有 approved 后才创建 session、写 `session.started`、安装事件 monitor 和显示录制控制条。
- denied / cancelled 不创建 session，MCP 返回 error result；错误文案尽量对齐官方已观察字符串。
- 自动化和 smoke 默认可配置为 auto-approve，避免本地 CI 因 UI 阻塞。

第二阶段再接 macOS OCU 自有开始确认 UI：

- UI 由 OCU runtime 绘制，不依赖 Codex.app。
- 确认 UI 和录制中控制条是两个阶段：前者决定是否开始，后者负责 Done / Discard。
- 如果宿主支持 MCP elicitation，可以在 gate 内优先走 MCP elicitation；宿主不支持或独立 CLI 使用时回退 OCU 自有 UI / policy。

验收点：

- approval approved / denied / cancelled 均有单测。
- denied / cancelled 时没有 session 目录和 active session。
- interactive 手工 smoke 能看到 OCU 自有开始确认，确认后才开始记录事件。
- MCP tools/list 不因 approval 增加任何参数或工具。

## 风险

- 官方 JSON schema 仍需要实际样本校准，不能只靠 strings 推断字段。
- AX diff 的 element identity、removed range summary 和 line budget 可能影响 replay skill 生成质量。
- 录制会涉及 Accessibility、Screen Recording、Input Monitoring 等系统权限，权限缺失或 input monitor 安装失败时必须给出可恢复状态；当前 baseline 会用低敏 `debug.error` 标记 input monitor 完全不可用。
- 截图采集有隐私和体积风险，需要默认最小化并保留 suppressed 记录。
- 浮层窗口可能被不同 Space、全屏应用或多显示器场景影响，需要单独 smoke。

### P0：官方 golden recordings

目标是把官方 observable behavior 固化成可比较样本，而不是继续只靠 `strings` 推断。

- 使用当前官方版本录制最小 stop / cancel / repeat start / no active stop / status 样本。
- 如需拉取更新版本，应保留版本号、采集日期、系统版本和脱敏说明，不覆盖 1.0.857 baseline。
- 样本至少覆盖 `events.jsonl`、`metadata.json` 或 `session.json`、`suppressed.jsonl`、MCP start/status/stop response。
- 样本保存前要脱敏 app title、用户输入、窗口内容和截图；无法脱敏的原始数据不进仓库。
- 官方样本入库使用 `scripts/import-event-stream-fixture.py <session-dir-or-metadata> --name <fixture-name> --source official --official-plugin-version ... [--mcp-transcript <probe-output.json>]`，默认把已知 metadata path 改成 fixture 相对路径，替换用户文本 / 标题 / timestamp / session id / app attribution，并移除 screenshot path；对于事件或 suppressed event 里的通用 `path` / 未知 `*Path` 字符串只保留 redacted marker 和长度，避免本机路径通过非标准字段漏进 fixture。如果传入 MCP probe output，会额外生成脱敏 `mcp-transcript.json`，保留 `event_stream_start/status/stop` response shape 作为官方对外行为证据。
- 样本落盘后先用 `open-computer-use event-stream validate --json <session-dir-or-metadata>` 或 `scripts/validate-event-stream-recording.py <session-dir-or-metadata>` 做结构检查；OCU 自身产物用 `--strict-ocu`，官方样本先不启用该开关，避免把 OCU alias/index 约束误当成官方事实。独立 skill/plugin 集成优先调用 runtime CLI，源码脚本只作为仓库内交叉检查。
- 结构检查通过后，用 `scripts/check-event-stream-golden-readiness.py <fixture-dir> --require-fixture-manifest --require-source official --require-official-plugin-version ... --require-session-alias --require-metadata-counts --require-handoff-paths --require-mcp-transcript --require-mcp-tool-response event_stream_start --require-mcp-tool-response event_stream_status --require-mcp-tool-response event_stream_stop` 判定样本是否足够作为动作类 golden；默认要求 final state、同时存在的 `metadata.json` / `session.json` 内容一致、已声明的 `eventCount` / `suppressedEventCount` 与文件行数一致、已声明的 handoff path 指向存在文件、`session.started` 恰好一次且为首事件、`session.ended` 恰好一次且为最终事件、至少一个动作事件和 AX payload，`--require-session-alias` 进一步要求两份 alias 文件都存在，`--require-metadata-counts` 进一步要求两项 count 字段都存在，`--require-handoff-paths` 进一步要求 fixture metadata 中的 `metadataPath` / `eventsPath` / `sessionPath` / `suppressedEventsPath` 四个字段都存在并可解析。注意这四个字段是脱敏 fixture 的归一化合同，不要求 official hosted 原始 JSON 同时返回 `suppressedEventsPath`；官方最小包可只有 `eventsPath` 和 `metadataPath` / `sessionPath`，导入器会补空 `suppressed.jsonl`。若样本要校准 AX / screenshot / 压缩 fallback，可额外要求 `--require-suppressed-events --require-suppressed-event-type <type>`，确保 suppressed evidence 不只停留在文件里。若样本来自 raw probe 并要校准 one-active、idempotent stop 或 completed status，可额外要求 `--require-mcp-response-shape repeatStartResponseShape --require-mcp-response-shape repeatStopResponseShape --require-mcp-response-shape finalStatusResponseShape`，确保这些 completed lifecycle 证据没有只停留在 transcript 里。tool 级 `--require-mcp-tool-response event_stream_start` 表示该 tool 至少有一次 result/error response；repeat shape 的 timeout 不会覆盖已成功的 start tool 证据，具体 repeat 语义必须用 `--require-mcp-response-shape` 单独要求。no-active status/stop 等生命周期样本需要显式加 `--allow-no-action --allow-no-ax-payload`，并按场景只要求对应 MCP response，避免把非动作样本误当成可校准事件语义的 golden。若 transcript 中 required tool 或 response shape 只有 timeout shape，readiness gate 会明确报 `MCP tool timed out` 或 `MCP response shape timed out`，该样本只能作为宿主外边界证据，不能当成成功 response golden。
- 官方样本与 OCU 候选录制的差异用 `scripts/compare-event-stream-recordings.py <official-fixture> <ocu-recording> --require-same-event-sequence --require-same-metadata-keys --require-same-metadata-values --require-handoff-paths --require-final-session-evidence --require-same-schema --require-semantic-fields --require-ax-diff-evidence` 量化；校准过程中可以先去掉 require 开关只看 JSON diff。`--require-same-metadata-keys` 守住 metadata/session 文件布局键；`--require-same-metadata-values` 守住 `state`、`active`、`endReason`、`eventCount`、`suppressedEventCount` 这类稳定状态值，避免 cancel / stop / count 漂移只因 key set 一致而漏掉；`--require-handoff-paths` 不比较路径字符串本身，但要求 baseline 与 candidate 都声明 `metadataPath` / `eventsPath` / `sessionPath` / `suppressedEventsPath` 且解析后文件存在，避免 candidate 保留 key 却丢掉可消费路径；`--require-final-session-evidence` 因兼容性保留原 flag 名，但现在守住 start/end lifecycle evidence：`session.started` 个数、首事件位置、`session.ended` 个数、最后事件位置和 endReason 集合，避免录制边界丢失；`--require-semantic-fields` 只盯 replay-critical 字段，后续官方 fixture 入库后可在不要求完整 schema 一字不差的情况下，先保证 target AX、drag 起止点、keyboard 目标、AX treeLines / cumulative treeLines 等关键证据没有丢失。若官方样本包含 `suppressed.jsonl` 降级 / 压缩 / 截图失败证据，可额外打开 `--require-same-suppressed-event-sequence --require-same-suppressed-schema`，要求 candidate 保留 suppressed stream 的事件顺序和 schema。若官方 fixture 和 OCU candidate 都带 `mcp-transcript.json`，可额外打开 `--require-mcp-response-shapes --require-same-mcp-response-schema`，要求 candidate 保留 baseline 中出现的 start/repeat/status/stop/final response shape、result/error/timeout 状态和 response schema。脱敏导入官方 fixture 时会保留 `treeLines` / `cumulativeTreeLines` 的 `+` / `-` / `~` marker，但移除原始文本，保证 AX diff 结构可比较且不泄漏内容。
- 采集前可先运行 `scripts/probe-event-stream-recording.py --target official` 确认官方 bundled client 的 non-recording surface 可用；如果要尝试 raw start/status/stop，用 `scripts/probe-event-stream-recording.py --target official --start-stop --timeout <seconds> --output <probe-output.json>`，脚本会记录 transcript，并在 start 返回或超时时尝试 status 和 stop。当前本机宿主外官方 raw start/status/stop 仍超时，因此不能把该路径当作 golden recording 采集成功条件。

### P1：官方兼容层校准

目标是让 `open-computer-use event-stream mcp` 对外继续保持官方三件套，同时把字段、状态流转和错误语义尽量对齐官方。

- MCP surface 只保留 `event_stream_start`、`event_stream_status`、`event_stream_stop` 三个无参数 tools。
- 校准 start/status/stop 返回 JSON 的字段名、嵌套结构、active/completed 状态和 idempotent stop 行为；no-active status/stop 已按 Codex-hosted 官方观察校准为 `isRecording=false` / `maxDurationSeconds=1800` 的最小 text JSON。
- 校准正常 stop、控制条 cancel、重复 start、无 active session stop、录制进程不可用等 endReason。
- 已补 30 分钟最长录制限制；官方二进制字符串只确认 “up to 30 minutes” 和内部 timeout task，未确认对外 endReason，因此 OCU baseline 暂用 `recording_time_limit_reached` 并标记待 official golden 校准。

### P1.5：Start approval 与独立 UI

目标是复刻官方 start 前 approval 的可观察语义，同时满足 OCU 脱离 Codex.app 独立运行的要求。

- 在 `EventStreamService.start()` 前加内部 approval gate，沉淀 approved / denied / cancelled 三种结果。已完成 baseline。
- denied / cancelled 使用官方已观察错误文案，且不创建 session。已完成 baseline。
- 测试、smoke 和 headless 路径必须可 auto-approve，避免阻塞自动化。已完成 baseline，使用 `OPEN_COMPUTER_USE_EVENT_STREAM_START_APPROVAL=approve`。
- macOS interactive runtime 接 OCU 自有开始确认 UI；用户确认后再显示录制中控制条。已完成 NSAlert baseline，文案会告知当前录制时限，默认 30 分钟；后续可替换成更贴近官方的浮层样式。
- MCP elicitation 作为 host-capability 优化，不作为独立 OCU 的硬依赖；baseline 已接入 `OPEN_COMPUTER_USE_EVENT_STREAM_START_APPROVAL=mcp|elicitation`，host 声明 `capabilities.elicitation` 时发送内部 `elicitation/create`，accept 才创建 session，decline / cancel / unsupported 均不创建 session。

### P2：事件和 AX payload 校准

## 验收门槛

进入独立 repo / skill 集成前，至少满足：

- `tools/list` 与官方当前 baseline 的 tool name、description、input schema、annotations 对齐。
- OCU golden tests 能用官方样本约束 session 文件布局、MCP response 和关键事件 schema。
- OCU smoke 能稳定覆盖 start/status/stop、session alias、AX payload、可选截图上下文和 `wait` listener。
